# Production Update: File Ownership Knowledge Upgrade

## Overview
We have completely overhauled the `FileOwnershipEvaluator` to mitigate three critical "Knowledge Traps" that previously distorted code ownership scores. Ownership is no longer a naive proxy for volume or touch frequency. Instead, it is rigorously tied to genuine cognitive context acquisition, actively mitigating data pollution from automated formatters, pairing sessions, and refactoring efforts.

## 1. Trap 1: The "Janitor" Fallacy (Unfiltered Churn)
**The Trap:** When a developer runs a codebase-wide linter, auto-formatter, or dependency upgrade (e.g., modifying thousands of lines across hundreds of files), naive evaluators instantly classify that developer as the dominant "Subject Matter Expert" for those files, overriding the developers who actually authored the business logic.
**The Fix:**
- The evaluator now incorporates the global `_get_file_weight()` mechanism.
- Commits that mutate lockfiles (e.g., `package-lock.json`, `poetry.lock`), binaries, and autogenerated files receive a strict weight of `0.0`.
- File renames are also completely ignored (`0.0`).
- This completely nullifies the "Janitor" anomaly. Mass-formatting trivial configurations will no longer dilute the true ownership scores of senior engineers.

## 2. Trap 2: Invisible Knowledge Transfer (The Silo Bug)
**The Trap:** Pair programming is the most effective way to spread architectural knowledge. However, if Developer A navigates and Developer B types, standard evaluators credit 100% of the knowledge acquisition to Developer B, creating a false "Knowledge Silo" alert in the Causal Engine.
**The Fix:**
- **Regex Extraction:** The evaluator extracts all secondary authors using the `Co-authored-by: Name <email>` Git trailer standard.
- **Fractional Distribution:** The calculated absolute churn score is mathematically divided by the total number of collaborators (`N`).
- By distributing the ownership points equally across the primary and secondary authors, the system correctly records that knowledge was shared dynamically, eliminating false key-person dependency warnings.

## 3. Trap 3: The Deletion Paradox (Negative Knowledge)
**The Trap:** Legacy algorithms often only reward "additions" or treat a single commit as a static `+1.0` touch. If an engineer spends two weeks refactoring a critical component and safely deletes 500 lines of spaghetti code, they might receive a lower ownership score than an intern who added 50 lines of boilerplate.
**The Fix:**
- The evaluator has abandoned the static `1.0` emission.
- Ownership is now directly derived from **absolute effective churn** (`(additions + deletions) * weight`).
- Safely deleting code requires profound context. The updated logic rewards refactoring by ensuring deletions positively compound the developer's ownership score.
- The downstream evidence synthesis layer will now correctly aggregate `effective_churn_by_author / total_effective_churn` to derive a precise majority-author percentage for each file.

*(Note: Per architectural directives for active MVP development, the `logic_version` of the evaluator remains at `v1.0.0`).*
