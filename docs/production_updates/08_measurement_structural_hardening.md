# Production Update: Measurement Structural Hardening (Registry & Versioning)

## Overview
This update resolves the "Metric Drift Trap" in the Measurement Layer by transitioning from hard-coded static dependencies to a decoupled, dynamically injected architecture. It guarantees that all measurements generated by the engine accurately reflect the exact version of the logic used, preventing un-tracked historical data corruption on dashboards.

## Key Changes

### 1. Evaluator Interface Versioning
- **Implementation**: Updated the `MeasurementEvaluator` Protocol in `app/measurement/core/interfaces.py`.
- **Enforcement**: Required every evaluator to explicitly implement `@property logic_version` and `@property metric_name`. This makes it impossible for developers to introduce new metrics without explicitly declaring the logic version and lineage.
- **Propagation**: Updated all 6 core evaluators (`ChangeComplexityEvaluator`, `ChangeImpactEvaluator`, `DeveloperActivityEvaluator`, `FileActivityEvaluator`, `FileOwnershipEvaluator`, `SubsystemActivityEvaluator`) to satisfy the new interface, currently locking them to `v1.0.0`.

### 2. Active Evaluator Registry
- **Implementation**: Enhanced `MeasurementRegistry` in `app/measurement/domain/registry.py` from holding static definitions to acting as an active Dependency Injection container for instantiated evaluators.
- **Benefit**: Evaluators are dynamically tracked and requested at runtime, moving the architecture toward configuration-driven metric toggling.

### 3. Engine Decoupling & Resilience
- **Implementation**: Removed all hard-coded imports of evaluators from `MeasurementEngine` (`app/measurement/core/engine.py`).
- **Resilience**: Evaluator execution was wrapped in a strict `try/except` loop. If an individual evaluator crashes (e.g., due to a rare Abstract Syntax Tree edge case), the failure is isolated and logged, ensuring the engine degrades gracefully rather than halting the entire measurement pipeline.
- **Traceability**: The `logic_version` property is now explicitly intercepted and stamped onto the output `Measurement` domain object's `version` field.

### 4. Database-Agnostic Schema
- **Architectural Ruling**: In accordance with DDD principles, the `logic_version` is tracked tightly in the pure domain `Measurement` object rather than immediately writing a one-off SQLite schema. When the persistent `SQLiteMeasurementStore` is built, the schema will organically derive from the robustly versioned domain object.
